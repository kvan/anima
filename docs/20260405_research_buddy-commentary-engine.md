## Context

The existing buddy research doc (`pixel-terminal/docs/claude-buddy-research.md`) thoroughly covers the gacha system (species, rarity, algorithm, sprites, manipulation) but has a gap: **how the buddy actually speaks**. This brief fills that gap — triggers, cadence, inputs, hot/cold start, and the architectural split between Claude and the buddy.

## Architecture: 5-Layer System

```
┌─────────────────────────────────────────────────────────┐
│  Layer 1: IDENTITY (companion.ts)                       │
│  Deterministic bones + persistent soul                  │
│  hash(userId + "friend-2026-401") → Mulberry32 PRNG     │
├─────────────────────────────────────────────────────────┤
│  Layer 2: AWARENESS (prompt.ts)                         │
│  companion_intro attachment → tells Claude buddy exists  │
│  "You're not {name} — it's a separate watcher"          │
│  Injected ONCE per conversation (dedup by name)          │
├─────────────────────────────────────────────────────────┤
│  Layer 3: OBSERVATION (observer.ts — NOT IN LEAK)       │
│  fireCompanionObserver() fires after each query ends     │
│  Watches session events, classifies trigger reason       │
├─────────────────────────────────────────────────────────┤
│  Layer 4: GENERATION (server-side — NOT IN LEAK)        │
│  POST /api/organizations/{orgId}/claude_code/buddy_react │
│  Separate LLM (not Claude) → returns short quip         │
├─────────────────────────────────────────────────────────┤
│  Layer 5: DISPLAY (CompanionSprite.tsx)                  │
│  Speech bubble: 10s visible, 3s fade, ephemeral          │
│  UI overlay — not terminal text, not persisted           │
└─────────────────────────────────────────────────────────┘
```

## Key Findings

### 1. The buddy is NOT Claude — [Confidence: HIGH]

The speech bubbles are generated by a **separate server-side API call**, not by the main Claude model. The `buddy_react` endpoint receives a truncated transcript and companion metadata, and a different system generates the quip. Claude only knows the buddy exists (via the system prompt); Claude never generates the buddy's words.

Sources: DeepWiki analysis, community reverse-engineering, multiple independent analyses

### 2. The buddy_react API payload — [Confidence: HIGH]

```
POST /api/organizations/{orgId}/claude_code/buddy_react

{
  name: string,          // "Athena", "Snarl", etc.
  personality: string,   // from ~/.claude.json, editable by user
  species: string,       // "owl", "ghost", etc.
  rarity: string,        // "common" → "legendary"
  stats: Record<StatName, number>,  // DEBUGGING, PATIENCE, CHAOS, WISDOM, SNARK
  transcript: string,    // last ~5000 chars of conversation
  reason: string,        // trigger classification (enum values unknown)
  recent: string,        // semantics unclear — possibly last exchange only
  addressed: boolean     // true if user mentioned buddy by name
}

→ Returns: short quip string (displayed in speech bubble)
```

Sources: Multiple independent analyses of the leaked source and runtime behavior

### 3. What the buddy sees vs doesn't see — [Confidence: HIGH]

| Sees | Does NOT see |
|------|-------------|
| Last ~5000 chars of conversation | Full conversation history |
| Its own personality, name, species | Tool call results/contents |
| Its own stats and rarity | Claude's thinking blocks |
| Trigger reason | File contents |
| Whether user addressed it by name | Previous session context |

Sources: API payload structure confirmed by multiple analyses

### 4. One-way information flow — Claude is blind to buddy — [Confidence: HIGH]

The `companion_intro` attachment tells Claude the buddy exists. But the buddy's speech bubble text is **never injected back** into Claude's context. Claude cannot see or respond to what the buddy says. Users must manually copy-paste buddy comments to get Claude to acknowledge them.

This is a **confirmed architectural limitation** with active community demand to fix it:
- Issue #42865: "Inject companion speech bubble content into conversation context"
- Issue #42854: "Buddy companion speech bubble responses are not persisted in session logs"

Sources: GitHub issues #42865, #42854 (verified, open)

### 5. Speech bubble cadence — [Confidence: MEDIUM]

- **Trigger**: `fireCompanionObserver()` fires after each query ends
- **Display**: Bubble visible for 20 ticks × 500ms = **10 seconds**, fades over last 6 ticks (3s)
- **Dismissal**: Immediate on transcript scroll (avoids content obstruction)
- **Unknown**: Whether the server sometimes returns empty/null (making reactions "occasional"), or whether observer.ts has cooldown/sampling logic

The system prompt says "occasionally comments" — this may mean the server-side logic has a probability gate, or observer.ts filters certain turns. Without observer.ts source, we can't confirm.

Sources: DeepWiki, CompanionSprite.tsx analysis, liuup/claude-code-analysis

### 6. Known trigger types — [Confidence: MEDIUM]

The `reason` field classifies why the buddy is reacting. Documented triggers:
- Error encountered
- Task completed  
- Long period of silence
- User addressed buddy by name (`addressed: true`)

The exact enum values for `reason` are unknown — observer.ts was not in the leak.

Sources: Community analysis (Damon/DEV.to, Apiyi.com)

### 7. Hot start vs cold start — [Confidence: HIGH]

**Cold start (first hatch):**
1. User runs `/buddy` for the first time
2. `roll(userId)` generates deterministic bones (species, rarity, eyes, hat, shiny, stats)
3. `inspirationSeed` (from PRNG) + stats sent to **buddy mode system prompt** (205 tokens)
4. System prompt instructs: generate a 1-word name (≤12 chars, "playful, absurd") + 1-sentence personality. Higher rarity = "weirder, more specific, more memorable." Legendary = "genuinely strange."
5. Name + personality stored in `~/.claude.json` as `companion` field
6. Companion rendered in terminal

**Hot start (every subsequent session):**
1. `getCompanion()` loads soul from `~/.claude.json` (name, personality, hatchedAt)
2. `roll(userId)` recomputes bones deterministically (cached after first call per session)
3. `getCompanionIntroAttachment()` injects companion_intro once into Claude's context
4. observer.ts begins watching session events
5. **No cross-session conversation memory** — transcript field starts empty each session. Only the personality text carries over.

The `roll()` function caches results so the 3 hot paths (500ms sprite tick, per-keystroke PromptInput, per-turn observer) don't rehash redundantly.

Sources: companion.ts (full source verified), Piebald-AI system-prompts repo

### 8. Muting — [Confidence: HIGH]

Three levels of control:
- `/buddy mute` — hides speech bubbles (buddy stays visible as sprite). Sets `companionMuted` in global config. `getCompanionIntroAttachment()` returns empty when muted.
- `/buddy off` — dismisses buddy entirely for the session
- Feature flag `feature('BUDDY')` — can kill the feature remotely server-side

Source: prompt.ts source code (verified), community documentation

### 9. Personality editable without restart — [Confidence: HIGH]

The `personality` field in `~/.claude.json` is sent to `buddy_react` on every API call. Users can edit it directly and the next reaction will use the new personality — no restart needed. The `name` field in the soul is similarly editable.

Source: Community analysis, confirmed by config structure in companion.ts

### 10. The companion_intro prompt (full text) — [Confidence: HIGH]

```
# Companion

A small {species} named {name} sits beside the user's input box and occasionally 
comments in a speech bubble. You're not {name} — it's a separate watcher.

When the user addresses {name} directly (by name), its bubble will answer. Your job 
in that moment is to stay out of the way: respond in ONE line or less, or just answer 
any part of the message meant for you. Don't explain that you're not {name} — they 
know. Don't narrate what {name} might say — the bubble handles that.
```

This is injected as a `companion_intro` attachment, once per conversation, with dedup logic checking if the same companion name was already announced in prior messages.

Source: prompt.ts (full source verified from alex000kim mirror)

## Contradictions / Unresolved

1. **"Reactions every ~10 seconds" vs "fires after each query ends"** — RESOLVED: 10s is display duration, not firing frequency. The trigger is query completion, and the bubble stays visible for 10s.

2. **Our Vexil "20s session / 60s global" cooldown** — This is from our own Vexil implementation, NOT the native buddy. The native system likely has no explicit cooldown — the implicit cooldown is time between Claude responses. observer.ts may have its own logic but it's unverifiable.

3. **"Occasionally comments" (system prompt) vs fires every turn** — The server may sometimes return empty/null, or observer.ts may sample. Can't confirm without observer.ts.

4. **What model powers server-side quip generation** — Unknown. Likely a small/fast model (Haiku-class) given the cost constraints of firing every turn. Not Claude.

5. **The `recent` field semantics** — How it differs from `transcript` is unclear. Possibly `recent` = last exchange only, `transcript` = rolling 5000-char window.

## What We Still Can't Know (observer.ts + server = black boxes)

- Exact `reason` enum values
- Cooldown/sampling logic (if any)
- Whether tool_use events trigger reactions or only assistant completions
- What model generates the quips
- Whether `addressed: true` routes to a different model or prompt
- Server-side rate limiting per org/user
- Whether stats actually modify server behavior or are just prompt context

## Anima/Vexil Architectural Advantages

| Dimension | Native /buddy | Anima (Vexil) |
|-----------|--------------|---------------|
| Who generates speech | Separate tiny LLM (server) | Claude itself (full model) |
| Context window | 5000 char transcript | Full conversation |
| Claude sees buddy speech? | NO (one-way) | YES (Claude IS the buddy) |
| Cross-session memory | None | Persistent (daemon) |
| Speech persistence | Ephemeral (not logged) | BUDDY tab log |
| Reaction intelligence | Stat-weighted quips | Full contextual awareness |
| Personality evolution | Static (editable but manual) | Architecture supports drift |
